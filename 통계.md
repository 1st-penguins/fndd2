# 통계 시스템 최적화 계획

## 현재 문제

통계를 볼 때마다 `attempts` 컬렉션 **전체**를 읽어서 클라이언트에서 계산하고 있음.
데이터가 쌓일수록 Firestore 읽기 비용이 선형 증가.

**예시**: attempts 50,000건 × 관리자 3명 × 하루 5회 조회 = **일 75만 reads**

---

## 통계를 사용하는 곳 (3곳)

| # | 소비처 | 위치 | 현재 방식 | 문제 |
|---|--------|------|-----------|------|
| 1 | 사용자 학습분석 | `analytics-dashboard.js` `loadAnalyticsData` | 본인 attempts 최대 3,000건 조회 | 상대적으로 적음 |
| 2 | 관리자 통계 탭 | `analytics-dashboard.js` `loadQuestionStatistics` | **questionStats 우선 조회로 수정 완료** | ✅ |
| 3 | 관리자 전용 통계 페이지 | `admin/statistics.html` `loadStatistics` | **매번 전체 attempts 직접 조회** | ❌ 핵심 낭비 |

---

## 방안 비교

### 방안 A: 클라이언트 증분 페칭 (사용자 아이디어)

> "조회 결과를 저장하고, 다음엔 추가된 것만 가져와서 더한다"

**장점**: 직관적, Cloud Function 변경 없음

**치명적 문제 — 완주 세션 필터와 비호환**:

```
시나리오:
1. 사용자가 20문제 중 15문제 풀고 중단
2. [T1] 통계 조회 → 이 15문제는 미완주이므로 제외됨 → 결과 저장
3. 사용자가 나머지 5문제 풀어 세션 완주
4. [T2] 증분 조회 → T1 이후의 5문제만 가져옴
5. ❌ 앞의 15문제는 이미 "미완주"로 제외된 상태
   → 세션이 완주됐지만 증분 조회로는 이전 15문제를 복구할 수 없음
```

이 문제를 해결하려면 세션 상태 추적이 필요 → 단순함이라는 장점이 사라짐.

**기타 문제**:
- localStorage = 기기/브라우저 종속, 관리자 간 공유 불가
- Firestore 저장 = 쓰기 비용 + 동시 접근 정합성 관리 복잡
- 첫 조회 비용은 동일 (전체 reads)

### 방안 B: Cloud Function 증분 집계 ⭐ 추천

> 세션 완주 시점에 Cloud Function이 questionStats 컬렉션에 증분 업데이트

**장점**:
- 조회 시 questionStats 문서만 읽으면 됨 (수천 reads → 수십 reads)
- 서버에 공유 데이터 → 모든 관리자가 즉시 혜택
- 완주 세션 필터 완벽 호환 (완주 시점에만 집계)
- **80% 이미 구현됨** (onNewAttempt, questionStats, 마이그레이션, 인덱스)

**단점**: Cloud Function 복잡도 약간 증가

### 방안 C: 스케줄 Function으로 매일 재집계

- 매일 전체 재계산 → 정확하지만 비용 절감 효과 적음
- 일 50,000 reads (재집계) + 조회 비용 = 여전히 비쌈
- 실시간성 떨어짐 (최대 24시간 지연)

---

## 비용 비교 (일일 기준)

가정: attempts 50,000건, 문제 500개, 일일 신규 500건, 관리자 3명 × 5회 조회

| 방안 | 일일 읽기 | 일일 쓰기 | 월간 읽기 |
|------|-----------|-----------|-----------|
| **현재** | 75,000 | 0 | 2,250,000 |
| **A (증분 페칭)** | ~5,500 | 0 | ~165,000 |
| **B (Cloud Fn)** | **3,000** | 500 | **90,000** |
| **C (매일 재집계)** | 53,000 | 500 | 1,590,000 |

**방안 B가 읽기 비용에서 압도적 우위**

---

## 최종 결론: 방안 B 완성

방안 A(사용자 아이디어)는 직관적이지만, 완주 세션 필터 때문에 이 프로젝트에선 부적합.
방안 B는 이미 대부분 구현되어 있어 **남은 작업이 적고 효과가 가장 큼**.

---

## 남은 작업

### 작업 1: 마이그레이션 실행
- `node scripts/migrate-question-stats.js`
- 기존 attempts → questionStats 초기 데이터 생성
- 1회성

### 작업 2: Cloud Function 배포
- `firebase deploy --only functions`
- onNewAttempt에 세션 완주 시 questionStats 증분 업데이트 이미 추가됨
- **추가 필요**: 저정답률 세션(30% 미만) 제외 필터

### 작업 3: admin/statistics.html을 questionStats 우선 조회로 변경
- 현재 매번 attempts 전체 쿼리 → questionStats 먼저 읽도록 수정
- analytics-dashboard.js의 loadQuestionStatistics 패턴 참조
- **핵심 절감 포인트** — 이것만 해도 비용 대폭 감소

### 작업 4: Firestore 인덱스 + Hosting 배포
- `firebase deploy --only firestore:indexes`
- `firebase deploy --only hosting`

### 작업 5 (선택): 주간 정합성 검증
- 스케줄 Function으로 주 1회 전체 재집계 vs questionStats 비교
- 차이가 크면 텔레그램 알림
- 우선순위 낮음 (보험)

---

## 배포 순서

```
1. node scripts/migrate-question-stats.js     ← 기존 데이터 집계
2. firebase deploy --only functions            ← 증분 업데이트 활성화
3. firebase deploy --only firestore:indexes    ← 복합 인덱스
4. firebase deploy --only hosting              ← 클라이언트 코드
```
